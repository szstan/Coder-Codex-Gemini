## 核心概念

### 1. 版本和里程碑

**版本 (Version)**：
- 定义：一个发布周期，包含多个里程碑
- 示例：v1.1.0（进度管理和优化版本）
- 状态：in_progress（开发中）、completed（已完成）

**里程碑 (Milestone)**：
- 定义：版本内的重要阶段，包含多个任务
- 示例：m1.1.2（进度管理系统）
- 完成度：通过任务完成数量计算

### 1.5. 时间字段说明

**时间格式标准**：
- 所有时间字段统一使用 **ISO 8601 格式**：`2026-01-20T15:30:00Z`
- UTC 时区（Z 后缀表示零时区）
- 精确到秒

**核心时间字段**：

| 字段名 | 含义 | 适用对象 | 示例 |
|--------|------|---------|------|
| `created_at` | 创建时间 | 任务、风险 | `2026-01-20T09:00:00Z` |
| `started_at` | 开始时间 | 任务、里程碑 | `2026-01-20T10:00:00Z` |
| `updated_at` | 最后更新时间 | 任务、风险 | `2026-01-20T15:30:00Z` |
| `due_date` | 截止时间 | 任务 | `2026-01-20T23:59:59Z` |
| `completed_at` | 完成时间 | 已完成任务 | `2026-01-20T18:00:00Z` |
| `completion_date` | 完成日期 | 里程碑 | `2026-01-19T18:00:00Z` |
| `resolved_at` | 解决时间 | 风险 | `2026-01-20T15:00:00Z` |

**时间字段用途**：
- **计算实际耗时**：`completed_at - started_at`
- **识别延期任务**：`当前时间 > due_date` 且状态非 completed
- **评估工作效率**：实际耗时 vs 预期耗时
- **追踪风险解决速度**：`resolved_at - created_at`

### 2. 任务管理

**任务状态**：
- `pending`：待开始
- `in_progress`：进行中
- `completed`：已完成
- `blocked`：被阻塞

**任务优先级**：
- 🔴 `high`：高优先级（紧急且重要）
- 🟡 `medium`：中优先级（重要但不紧急）
- 🟢 `low`：低优先级（可延后）

### 3. 风险和阻塞项

**风险 (Risk)**：
- 定义：可能影响进度的潜在问题
- 严重性：high（高）、medium（中）、low（低）
- 状态：monitoring（监控中）、mitigated（已缓解）、resolved（已解决）

**阻塞项 (Blocker)**：
- 定义：当前阻止任务继续的问题
- 必须立即解决，否则任务无法推进

## 使用场景

### 场景 1：查看项目总体进度

**何时使用**：
- 会话启动时想了解项目状态
- 向团队汇报进度
- 评估是否能按时交付

**操作步骤**：
```bash
# 在 Claude Code 中执行
/ccg-progress
```

**输出示例**：
```
📊 **项目进度概览**

**当前版本**: v1.1.0 (60% 完成)
**目标发布**: 2026-02-01 (剩余 12 天)

**里程碑进度**:
- ✅ 会话管理系统 (100%)
- 🚧 进度管理系统 (30%)
- ✅ Codex Review 模式 (100%)

**活跃任务** (5 个):
- 🔴 创建进度跟踪配置 (进行中)
- 🔴 创建进度管理 Skill (待开始)

**风险** (2 个):
- 🟡 进度数据结构设计复杂
```

### 场景 2：开始新任务

**操作步骤**：
1. 打开 `.ccg/progress.json`
2. 在 `active_tasks` 数组中添加新任务
3. 保存文件

**示例**：
```json
{
  "id": "task-006",
  "title": "实现自动进度报告生成",
  "priority": "medium",
  "status": "pending",
  "assignee": "Claude",
  "created_at": "2026-01-20",
  "due_date": "2026-01-24",
  "milestone": "m1.1.2",
  "tags": ["progress-management", "automation"]
}
```

### 场景 3：标记任务完成

**操作步骤**：
1. 打开 `.ccg/progress.json`
2. 找到对应任务，修改状态为 `completed`
3. 将任务从 `active_tasks` 移到 `completed_tasks`
4. 添加完成时间
5. 更新里程碑进度

**示例**：
```json
// 从 active_tasks 移除
// 添加到 completed_tasks
{
  "id": "task-001",
  "title": "创建进度跟踪配置",
  "completed_at": "2026-01-20",
  "milestone": "m1.1.2"
}
```

### 场景 4：识别和添加风险

**操作步骤**：
1. 打开 `.ccg/progress.json`
2. 在 `risks` 数组中添加风险项
3. 定义缓解措施

**示例**：
```json
{
  "id": "risk-003",
  "title": "第三方 API 可能不稳定",
  "severity": "high",
  "impact": "可能导致功能无法使用",
  "mitigation": "添加重试机制和降级方案",
  "status": "monitoring",
  "created_at": "2026-01-20"
}
```

- 已完成任务
- 风险和阻塞项
- 统计指标

**更新频率**：任务状态变化时实时更新

## 常见问题

**Q: 如何自动更新进度？**
A: 部分自动（任务完成时），部分手动（添加新任务）

**Q: 进度数据会丢失吗？**
A: 不会，所有数据保存在 `.ccg/progress.json` 中

**Q: 如何查看历史进度？**
A: 查看 `PROJECT_ROADMAP.md` 的变更历史部分

**Q: 可以自定义进度指标吗？**
A: 可以，编辑 `.ccg/progress.json` 的 `metrics` 字段
